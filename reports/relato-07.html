1) Introdução

Referência: FLUSP - Sugestões de Patches para o Kernel Linux (Sugestão 1)

Com base nas sugestões disponíveis no site do FLUSP, eu e minha dupla na disciplina (Felipe Ribeiro de Souza) decidimos seguir a Sugestão 1: Replace mutex_lock(&lock) and mutex_unlock(&lock) calls with guard(mutex)(&lock).

A ideia é substituir o uso manual de mutex_lock e mutex_unlock, que pode gerar dois problemas principais:

  1. O desenvolvedor pode esquecer de chamar mutex_unlock, principalmente em fluxos condicionais (como switch/case).
  2. Para contornar isso, muitos desenvolvedores usam o padrão goto error, centralizando o unlock no final.

Esse padrão típico é exemplificado assim:

int function_with_mutex_and_goto_error_pattern(void)
{
    int ret = 0;

    mutex_lock(&lock);
    switch (var) {
    case FOO:
        do_foo();
        break;
    case BAR:
        ret = do_first_bar();
        if (ret)
            goto out;

        ret = do_second_bar();
        if (ret)
            goto out;

        do_third_bar();
        break;
    default:
        ret = -EINVAL;
    }

out:
    mutex_unlock(&lock);
    return ret;
}

A solução é implementar a macro guard(mutex)(&lock), definida em docs.kernel.org/core-api/cleanup.html, que libera o mutex automaticamente ao sair do escopo (por qualquer caminho: return, fim da função, etc.). Assim, sem perda de funcionalidade, o código fica: 

int function_with_guard_mutex(void)
{
    int ret = 0;

    guard(mutex)(&lock);
    switch (var) {
    case FOO:
        do_foo();
        return 0;
    case BAR:
        ret = do_first_bar();
        if (ret)
            return ret;

        ret = do_second_bar();
        if (ret)
            return ret;

        do_third_bar();
        return 0;
    default:
        return -EINVAL;
    }
}

Após discussão com os monitores, optamos por seguir esse caminho.

3) Alterando mutex_lock

O método de mudança seguiu cinco passos simples:

  1. Localizar as funções que possuíam mutex_lock e mutex_unlock.
  2. Verificar se faz sentido aplicar o novo padrão.
  3. Verificar se apenas uma porção da função lida com o mutex (usar scoped_guard) ou se toda a função precisa (usar guard).
  4. Implementar a respectiva macro.
  5. Remover o mutex_lock e o mutex_unlock.

3.1) Usando scoped_guard em ad7280_store_balance_sw

A primeira função alterada foi ad7280_store_balance_sw. Como o mutex protegia apenas uma porção interna da função (e havia código antes dele), usamos scoped_guard:

/* Antes */
mutex_lock(&st->lock);
if (readin)
    st->cb_mask[devaddr] |= BIT(ch);
else
    st->cb_mask[devaddr] &= ~BIT(ch);

ret = ad7280_write(st, devaddr, AD7280A_CELL_BALANCE_REG, 0,
           FIELD_PREP(AD7280A_CELL_BALANCE_CHAN_BITMAP_MSK,
                  st->cb_mask[devaddr]));
mutex_unlock(&st->lock);
/* Depois */
scoped_guard(mutex, &st->lock) {
    if (readin)
        st->cb_mask[devaddr] |= BIT(ch);
    else
        st->cb_mask[devaddr] &= ~BIT(ch);

    ret = ad7280_write(st, devaddr, AD7280A_CELL_BALANCE_REG, 0,
               FIELD_PREP(AD7280A_CELL_BALANCE_CHAN_BITMAP_MSK,
                      st->cb_mask[devaddr]));
}

O mesmo padrão de scoped_guard foi aplicado às funções ad7280_show_balance_timer, ad7280_store_balance_timer e ad7280_read_raw.

3.2) Usando guard em ad7280a_write_thresh

Para a função ad7280a_write_thresh, que continha um switch/case com o padrão goto err_unlock, utilizamos guard(mutex), o unlock acontece automaticamente em qualquer return. Também foi necessário substituir os break intermediários por return: 

/* Antes */
mutex_lock(&st->lock);
switch (chan->type) {
case IIO_VOLTAGE:
    ...
    switch (dir) {
    case IIO_EV_DIR_RISING:
        ...
        if (ret)
            break;
        st->cell_threshhigh = value;
        break;
    ...
    default:
        ret = -EINVAL;
        goto err_unlock;
    }
    break;
...
default:
    ret = -EINVAL;
    goto err_unlock;
}
err_unlock:
    mutex_unlock(&st->lock);
    return ret;
/* Depois */
guard(mutex)(&st->lock);

switch (chan->type) {
case IIO_VOLTAGE:
    ...
    switch (dir) {
    case IIO_EV_DIR_RISING:
        ...
        if (ret)
            return ret;
        st->cell_threshhigh = value;
        return 0;
    case IIO_EV_DIR_FALLING:
        ...
        if (ret)
            return ret;
        st->cell_threshlow = value;
        return 0;
    default:
        return -EINVAL;
    }
case IIO_TEMP:
    ...
    switch (dir) {
    case IIO_EV_DIR_RISING:
        ...
        if (ret)
            return ret;
        st->aux_threshhigh = value;
        return 0;
    case IIO_EV_DIR_FALLING:
        ...
        if (ret)
            return ret;
        st->aux_threshlow = value;
        return 0;
    default:
        return -EINVAL;
    }
default:
    return -EINVAL;
}

O que mudou em ad7280a_write_thresh:

Antes Depois
mutex_lock + mutex_unlock manual guard(mutex)(&st->lock) = unlock automático
goto err_unlock para todos os erros return -EINVAL direto, sem rótulo
err_unlock: + mutex_unlock + return ret no final Removidos completamente
ret usada em todos os caminhos ret só onde ad7280_write pode falhar

8) Desenvolvimento v2

Em menos de 24 horas recebi uma réplica do revisor Andy Shevchenko apontando três problemas:

  1. O conteúdo dentro de scoped_guard(mutex, &st->lock) { ... } precisava ser reindentado um nível para a direita (tab extra).
  2. Em funções onde o mutex cobre toda a lógica restante, guard() seria preferível ao scoped_guard() por legibilidade.
  3. Uma linha de continuação de argumento dentro de scoped_guard estava com indentação incorreta.

Trecho do review (resumido):

> -	mutex_lock(&st->lock);
> +	scoped_guard(mutex, &st->lock) {

All in this scope has to be reindented by one tab level up (to the right).

> +		scoped_guard(mutex, &st->lock) {

Has to be reindented. {} are not needed.

But why not simply guard()()?
-- Andy Shevchenko

Seguindo as sugestões, corrigi os erros de indentação (estava usando espaços em vez de tabs em alguns lugares) e reavaliando o código, troquei os lugares onde usava scoped_guard por guard para melhor legibilidade.

9) Desenvolvimento v3

Andy Shevchenko respondeu novamente, apontando (espero) a última correção necessária. O problema estava em ad7280_show_balance_timer: 

> -	mutex_lock(&st->lock);
> +	guard(mutex)(&st->lock);

This will add some rather heavy operation into the critical section.
Why do we need that? Please, use scoped_guard() in such cases.

> -	mutex_unlock(&st->lock);
>
>  	if (ret < 0)
>  		return ret;
-- Andy Shevchenko

O revisor tem razão: nessa função o mutex protege apenas a leitura do registrador; o processamento posterior (FIELD_GET, sysfs_emit) não precisa estar na seção crítica. Na v1 eu já havia feito exatamente isso com scoped_guard, mas ao seguir o feedback da v2 de priorizar guard, acabei expandindo desnecessariamente a seção crítica. A solução é reverter para a abordagem da v1, agora com a indentação correta: 

static ssize_t ad7280_show_balance_timer(struct iio_dev *indio_dev,
					 uintptr_t private,
					 const struct iio_chan_spec *chan,
					 char *buf)
{
	struct ad7280_state *st = iio_priv(indio_dev);
	unsigned int msecs;
	int ret;

	scoped_guard(mutex, &st->lock)
		ret = ad7280_read_reg(st, chan->address >> 8,
					(chan->address & 0xFF) + AD7280A_CB1_TIMER_REG);

	if (ret < 0)
		return ret;

	msecs = FIELD_GET(AD7280A_CB_TIMER_VAL_MSK, ret) * 71500;

	return sysfs_emit(buf, "%u.%u\n", msecs / 1000, msecs % 1000);
}
