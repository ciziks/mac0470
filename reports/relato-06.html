1) Introdução ao IIO

Referências: FLUSP - Anatomia do iio_dummy, FLUSP - Experimento com iio_dummy

O subsistema Industrial I/O (IIO) é responsável por gerenciar dispositivos que produzem ou consomem sinais analógicos, como sensores de temperatura, acelerômetros, giroscópios e magnetômetros. O módulo iio_dummy é um exemplo educacional de Device Driver dentro desse subsistema.

Um conceito central do IIO é o canal (channel): cada canal representa um fluxo de dados de um dispositivo. Um único dispositivo IIO pode ter um ou mais canais. A configuração de cada canal é feita através de uma struct iio_chan_spec, declarada em include/linux/iio/iio.h. Essa struct é flexível o suficiente para descrever diferentes tipos de dispositivos com a mesma estrutura base.

2) Habilitando o iio_dummy

O primeiro passo foi habilitar o iio_dummy no menuconfig do kernel: 

make -C "$IIO_TREE" menuconfig

Após ativar o iio_dummy, confirmei a configuração gerada no .config: 

#
# IIO dummy driver
#
CONFIG_IIO_DUMMY_EVGEN=m
CONFIG_IIO_SIMPLE_DUMMY=m
CONFIG_IIO_SIMPLE_DUMMY_EVENTS=y
CONFIG_IIO_SIMPLE_DUMMY_BUFFER=y
# end of IIO dummy driver

3) Compilar, Load e Unload do iio_dummy

Para garantir que o kw build estava corretamente configurado com cross-compile para arm64: 

(LK-DEV) ciziks-debian@ciziks:/home/lk_dev/iio$ kw config --show build
[LOCAL ] build.arch=arm64
[LOCAL ] build.kernel_img_name=Image.gz
[LOCAL ] build.cross_compile=aarch64-linux-gnu-
[LOCAL ] build.menu_config=nconfig
[LOCAL ] build.doc_type=htmldocs
[LOCAL ] build.cpu_scaling_factor=100

Rodei o kw build (ao invés do make M=... sugerido pelo tutorial, pois esse último usa o gcc da máquina host, o que pode causar problemas com cross-compilation para arm64).

O kw deploy --modules apresentou o mesmo erro de incompatibilidade de formato de arquivo que havia ocorrido anteriormente. Para contornar, enviei apenas os arquivos .ko do módulo diretamente para a VM: 

kw ssh --send drivers/iio/dummy/iio_dummy.ko
kw ssh --send drivers/iio/dummy/iio_dummy_evgen.ko

Dentro da VM, instalei os módulos manualmente:

mkdir -p /lib/modules/$(uname -r)/kernel/drivers/iio/dummy
cp iio_dummy.ko /lib/modules/`uname -r`/kernel/drivers/iio/dummy
cp iio_dummy_evgen.ko /lib/modules/`uname -r`/kernel/drivers/iio/dummy
depmod --quick

4) Explorando o iio_dummy via configfs

O configfs é um sistema de arquivos virtual que permite criar e configurar objetos do kernel em tempo de execução através de operações de diretório. Para montá-lo: 

sudo mkdir /mnt/iio_experiments/
sudo mount -t configfs none /mnt/iio_experiments/
root@localhost:~# ls /mnt/iio_experiments/
iio  pci_ep

Embora o tutorial não deixe explícito, o diretório iio/devices/ só existe enquanto o módulo estiver carregado. Após rodar modprobe iio_dummy: 

root@localhost:~# ls /mnt/iio_experiments/iio/devices/
dummy

Criando um novo dispositivo virtual:

sudo mkdir /mnt/iio_experiments/iio/devices/dummy/my_glorious_dummy_device
root@localhost:~# ls -l /sys/bus/iio/devices/
total 0
lrwxrwxrwx 1 root root 0 Apr  6 20:34 iio:device0 -> ../../../devices/iio:device0
lrwxrwxrwx 1 root root 0 Apr  6 20:34 iio_evgen -> ../../../devices/iio_evgen

O simples ato de criar um diretório no configfs instanciou um novo dispositivo IIO (iio:device0) no sysfs.

6) Modificando o Canal para Bússola 3-eixos

O experimento propõe adicionar suporte a um magnetômetro de 3 eixos (X, Y, Z) ao iio_simple_dummy. As modificações foram feitas em dois arquivos: drivers/iio/dummy/iio_simple_dummy.h e drivers/iio/dummy/iio_simple_dummy.c.

6.1) Alterações no header (iio_simple_dummy.h)

Adicionei a constante para o número de eixos e o buffer na struct de estado:

/* No início do arquivo */
#define DUMMY_AXIS_XYZ 3

/* Na struct iio_dummy_state */
u16 buffer_compass[DUMMY_AXIS_XYZ];

Adicionei os novos índices de scan no enum iio_simple_dummy_scan_elements:

DUMMY_INDEX_SOFT_TIMESTAMP,
DUMMY_MAGN_X,
DUMMY_MAGN_Y,
DUMMY_MAGN_Z,

6.2) Adicionando canais em iio_simple_dummy.c

No array iio_dummy_channels[], adicionei três entradas para os eixos X, Y e Z do magnetômetro ao final da struct: 

{
    .type = IIO_MAGN,
    .modified = 1,
    .channel2 = IIO_MOD_X,
    .info_mask_separate = BIT(IIO_CHAN_INFO_RAW),
    .info_mask_shared_by_type = BIT(IIO_CHAN_INFO_SCALE),
    .scan_index = DUMMY_MAGN_X,
    .scan_type = {
        .sign = 'u',
        .realbits = 16,
        .storagebits = 16,
        .shift = 0,
    },
},
{
    .type = IIO_MAGN,
    .modified = 1,
    .channel2 = IIO_MOD_Y,
    .info_mask_separate = BIT(IIO_CHAN_INFO_RAW),
    .info_mask_shared_by_type = BIT(IIO_CHAN_INFO_SCALE),
    .scan_index = DUMMY_MAGN_Y,
    .scan_type = {
        .sign = 'u',
        .realbits = 16,
        .storagebits = 16,
        .shift = 0,
    },
},
{
    .type = IIO_MAGN,
    .modified = 1,
    .channel2 = IIO_MOD_Z,
    .info_mask_separate = BIT(IIO_CHAN_INFO_RAW),
    .info_mask_shared_by_type = BIT(IIO_CHAN_INFO_SCALE),
    .scan_index = DUMMY_MAGN_Z,
    .scan_type = {
        .sign = 'u',
        .realbits = 16,
        .storagebits = 16,
        .shift = 0,
    },
},

O tutorial também indicou trocar o número fixo 4 pelo símbolo correto em IIO_CHAN_SOFT_TIMESTAMP: 

IIO_CHAN_SOFT_TIMESTAMP(DUMMY_INDEX_SOFT_TIMESTAMP)

6.3) Inicializando os valores no iio_dummy_init_device

st->buffer_compass[0] = 78;
st->buffer_compass[1] = 10;
st->buffer_compass[2] = 3;

6.4) Expandindo o read_raw

Adicionei o novo caso IIO_MAGN em __iio_dummy_read_raw para expor os dados ao user space: 

case IIO_MAGN:
    switch(chan->scan_index) {
        case DUMMY_MAGN_X:
            *val = st->buffer_compass[0];
            break;
        case DUMMY_MAGN_Y:
            *val = st->buffer_compass[1];
            break;
        case DUMMY_MAGN_Z:
            *val = st->buffer_compass[2];
            break;
        default:
            *val = 99;
            break;
    }
    return IIO_VAL_INT;

E adicionei o suporte ao scale do magnetômetro em IIO_CHAN_INFO_SCALE:

case IIO_MAGN:
    *val = 0;
    *val2 = 2;
    ret = IIO_VAL_INT_PLUS_MICRO;
    break;

6.5) Atualizando a descrição do módulo

MODULE_DESCRIPTION("IIO dummy driver -> IIO dummy modified by Lucas Ciziks");

Não foi necessário alterar o iio_dummy_write_raw, pois ele apenas escreve diretamente na struct de estado, sem distinção por tipo de canal.
