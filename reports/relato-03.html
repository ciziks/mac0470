1) Introdução

Referência: FLUSP - Introduction to Linux kernel build configuration and modules

Este tutorial cobre o ciclo completo de desenvolvimento de um módulo do kernel Linux: criação do código-fonte, integração com o sistema de configuração (KConfig) e o sistema de build (Kbuild), habilitação via menuconfig, compilação cruzada para ARM64, deploy na VM e carregamento/descarregamento dinâmico do módulo.

Ao final, também são apresentados exercícios sobre configurações de módulos com allmodconfig, localmodconfig e habilitação do IKCONFIG para inspecionar o .config de dentro da VM.

O primeiro passo foi criar o arquivo simple_mod.c dentro do diretório drivers/misc/ da árvore IIO. O módulo define funções de init e exit que imprimem mensagens no log do kernel via pr_info, registradas com as macros module_init e module_exit.

Para instalar os módulos compilados na VM, precisei montar a imagem com guestmount. Na tentativa inicial encontrei um erro de crash no libguestfs:

Após troubleshooting com minha dupla (Felipe), que apresentou o mesmo problema, identificamos duas causas: a VM estava em execução e o dispositivo utilizado era sda2 em vez de vda2. Desliguei a VM e corrigi o comando:

7) Exercícios

Exercício 1 - allmodconfig vs localmodconfig

O primeiro exercício propõe comparar o resultado de make allmodconfig (habilita todos os módulos possíveis) com make localmodconfig (mantém apenas os módulos carregados na máquina atual).

Rodei make allmodconfig e instal os módulos na VM. 

make allmodconfig
kw ssh --get '~/vm_mod_list'

Em seguida, apliquei make localmodconfig. O .config reduziu drasticamente: de ~50 módulos para apenas 6. Isso impactou diretamente o tempo de compilação, resultando em um build significativamente mais rápido.

Exercício 2 - Habilitando IKCONFIG

O segundo exercício propõe habilitar a opção CONFIG_IKCONFIG e CONFIG_IKCONFIG_PROC para disponibilizar o .config de dentro da VM via /proc/config.gz.

Desliguei a VM, abri o menuconfig e habilitei a opção: 

make menuconfig

Após o build e iniciar a VM, o arquivo estava disponível em /proc/config.xz:

kw build
sudo virsh start arm64
kw ssh

Exercício 3 - KBUILD_MODNAME como prefixo de log

O terceiro exercício propõe adicionar a macro pr_fmt para prefixar automaticamente todas as mensagens de log com o nome do módulo: 

#define pr_fmt(fmt) KBUILD_MODNAME ": " fmt

Adicionei essa linha acima dos includes no simple_mod.c, conforme sugerido pelo tutorial, e recompilei o módulo: 

kw build

Para enviar apenas o módulo atualizado à VM sem reinstalar tudo: 

kw ssh --send drivers/misc/simple_mod.ko

Tentei recompilar o simple_mod_part.c da mesma forma, mas nenhuma das tentativas - nem kw build nem modules_prepare - gerou o .ko correspondente para reenviar à VM.

De qualquer forma, para o simple_mod o resultado foi correto. Após copiar o módulo e rodar depmod --quick, os logs passaram a exibir o prefixo simple_mod::

Mesmo não tendo conseguido replicar ao simple_mod_part, o exercício foi concluído com sucesso para o simple_mod.
