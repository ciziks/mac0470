5) Enviando Patches por Email com o Git e kw send-patch

Configuração do git send-email com ProtonMail via SMTP e do kw send-patch e teste prático de envio de patches por email com ambas as ferramentas.

git patches git-send-email kw send-patch smtp kernel

1) Introdução

Referência: FLUSP - Sending patches by email with Git

Antes de iniciar o tutorial, li o artigo Patch Philosophy do KernelNewbies para entender o que é um patch e as boas práticas a serem adotadas ao enviá-lo. O artigo aborda pontos essenciais como manter patches pequenos e focados, escrever mensagens de commit descritivas e seguir o fluxo de revisão da lista de emails do kernel.

Este tutorial, embora curto, é fundamental para o fim do ciclo kernel, quando precisaremos enviar nosso patch de contribuição para o subsistema IIO.

2) Instalação do git-email

O git send-email não vem habilitado por padrão. É necessário instalar o pacote git-email separadamente: 

sudo apt-get install git-email

3) Configurando o Git

Com o pacote instalado, configurei as informações de identidade globais do Git, usadas como remetente nos emails gerados pelo git send-email: 

git config --global user.name "Lucas Ivars Cadima Ciziks"
git config --global user.email "meu-email-pessoal"

4) Configurando o Servidor SMTP

O tutorial original demonstra a configuração com Gmail, mas como meu email principal é do ProtonMail, segui o guia de SMTP do ProtonMail e usei as configurações correspondentes: 

git config --global sendemail.smtpencryption tls
git config --global sendemail.smtpserver "smtp.protonmail.ch"
git config --global sendemail.smtpuser "<meu-email-pessoal>"
git config --global sendemail.smtpserverport "587"
git config --global sendemail.smtppass "<meu-token-gerado>"

O tutorial alerta sobre a insegurança do último comando: o token fica salvo em texto puro dentro do ~/.gitconfig. Por conveniência, optei por utilizá-lo mesmo assim.

Por fim, configurei o git send-email para não me incluir automaticamente em cópia nos próprios emails enviados: 

git config --global sendemail.suppresscc self

5) Opções do git send-email

O tutorial apresenta as formas básicas de invocar o comando: 

git send-email -N          # Envia os últimos N patches
git send-email ds2901..89e73  # Envia os commits do hash ds2901 até 89e73

Além disso, há diversas opções úteis:

  • --annotate: abre o email no editor de texto antes de enviar. Dentro do editor, é recomendado usar o separador --- (scissor separator) para adicionar comentários, especialmente em revisões de versão (v2, v3...) após um review.
  • --cover-letter: adiciona um email introdutório com uma breve explicação das modificações do patchset. Um exemplo real de patchset com cover letter pode ser visto em lore.kernel.org.
  • --thread: envia todos os emails do patchset na mesma thread.
    • --no-chain-reply-to: faz com que todos os emails respondam ao primeiro da thread, comportamento esperado ao contribuir com o kernel Linux.
    • --chain-reply-to (padrão): cada email responde ao anterior.
  • --dry-run: simula o envio sem de fato mandar os emails.
  • --to: destinatário principal do email.
  • --cc: destinatários em cópia.

Referência do comando: git-send-email docs

6) Testando o Envio

Para testar o fluxo completo, executei o comando dentro do repositório desta página web, enviando o patch mais recente para o meu próprio email com a flag --annotate para editar o conteúdo antes do envio: 

git send-email --annotate --to="<outro-email-pessoal>" -1

7) Continuação: Tutorial 6 - kw send-patch

Referência: FLUSP - Sending patches with Git and a USP email

Como o Tutorial 6 está no mesmo contexto do Tutorial 5 (ambos tratam de envio de patches por email), ele foi incorporado a este relato como continuação. O Tutorial 6 foca no uso do kw send-patch (ferramenta do kworkflow) para envio de patches.

8) Proxy de Email

O tutorial instrui rodar um proxy de email para resolver a autenticação do Gmail sem App Password. Como configurei um email com domínio próprio ao invés do email USP, não precisei rodar o proxy, pois a autenticação SMTP já estava configurada corretamente desde o Tutorial 5.

9) Testando o kw send-patch

Após o Tutorial 5, o git send-email já estava configurado corretamente. Basta agora testar o kw send-patch.

Para o teste, criei um arquivo LUCAS.CIZIKS.txt com o conteúdo I AM LUCAS CIZIKS. e adicionei uma exceção no .gitignore para aceitar esse arquivo. Em seguida, criei um commit com o mesmo nome.

Antes de enviar, utilizei a flag --simulate para verificar se a formatação estava correta: 

kw send-patch --send --private --simulate --to 'luciziks@hotmail.com' --to 'dsl26oauthproxy@gmail.com'

Confirmada a formatação, enviei o "patch" de fato aos remetentes. Uma pequena correção em relação ao tutorial original: ele usa dois --to, o que faz com que apenas o último destinatário receba o email. Corrigi usando --cc para o segundo destinatário: 

kw send-patch --send --private --to 'luciziks@hotmail.com' --cc 'dsl26oauthproxy@gmail.com'

Tutoriais 5 e 6 concluídos! Tanto o git send-email quanto o kw send-patch estão funcionando.
